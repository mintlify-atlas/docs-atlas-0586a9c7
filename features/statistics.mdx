## What Kutt tracks

Every time a visitor follows one of your short links, Kutt records a visit. The following data points are captured per visit:

| Data point | Details |
|---|---|
| **Browser** | Chrome, Edge, Firefox, IE, Opera, Safari, or Other |
| **Operating system** | Android, iOS, Linux, macOS, Windows, or Other |
| **Country** | Derived from the `CF-IPCountry` header (set by Cloudflare). Falls back to `xx` when the header is absent. |
| **Referrer** | The `Referer` request header, lowercased. Empty string when not present. |

Stats are aggregated by hour. Multiple visits within the same clock-hour are stored in a single visit row rather than as separate records.

---

## Bot filtering

Kutt uses the [`isbot`](https://github.com/nicolo-ribaudo/isbot) library to detect bot traffic based on the `User-Agent` header. Visits from known bots are **not** counted in your statistics and do not increment the link's `visit_count`.

<Note>
  Only links owned by a registered user track statistics. Links created anonymously (when `DISALLOW_ANONYMOUS_LINKS=false`) do not record visit data.
</Note>

---

## Time ranges

Statistics are available across four time ranges:

| Key | Covers | `views` array length |
|---|---|---|
| `lastDay` | Last 24 hours | 24 elements (one per hour) |
| `lastWeek` | Last 7 days | 7 elements (one per day) |
| `lastMonth` | Last 30 days | 30 elements (one per day) |
| `lastYear` | Last 12 months | 12 elements (one per month) |

Each element in the `views` array represents the total visit count for that period slot. The last element in the array is the most recent slot.

---

## Privacy

Statistics are private to the link owner. Only the user who created the link can view its stats. Admins can also access stats for any link on the instance.

---

## Accessing statistics

### Via the dashboard

Open your link dashboard, find a link, and click the statistics icon. The stats page shows charts for each time range along with the browser, OS, country, and referrer breakdowns.

### Via the API

```http
GET /api/v2/links/:id/stats
X-API-KEY: <your-api-key>
```

Replace `:id` with the UUID of the link (the `id` field returned when you created or fetched the link).

---

### Field reference

| Field | Type | Description |
|---|---|---|
| `lastDay.views` | `number[24]` | Hourly visit counts for the past 24 hours. Index 23 = most recent hour. |
| `lastWeek.views` | `number[7]` | Daily visit counts for the past 7 days. Index 6 = today. |
| `lastMonth.views` | `number[30]` | Daily visit counts for the past 30 days. Index 29 = today. |
| `lastYear.views` | `number[12]` | Monthly visit counts for the past 12 months. Index 11 = current month. |
| `*.total` | `number` | Sum of all views in that period. |
| `*.stats.browser` | `array` | Per-browser counts as `{ name, value }` pairs. |
| `*.stats.os` | `array` | Per-OS counts as `{ name, value }` pairs. |
| `*.stats.country` | `array` | Per-country counts as `{ name, value }` pairs. Country codes are ISO 3166-1 alpha-2, lowercased. |
| `*.stats.referrer` | `array` | Per-referrer domain counts as `{ name, value }` pairs. |

---

## Redis caching

When Redis is enabled (`REDIS_ENABLED=true`), stats responses are cached in Redis for 60 seconds. This means very recent visits may not appear in the API response immediately.
